# v0.5 ‚ÄĒ Paint engine + Selection toolset

Ship date: 2026-05-09. Headline: Tiramisu now paints and selects with the
core tools you'd expect from a Photoshop/Affinity peer ‚ÄĒ pencil, eraser,
free-form / polygonal / magic-wand / Vision-driven selections, and a
Photoshop-style ring cursor that follows the pointer.

## Pieces

### Rasterize Layer (`DocumentStore.rasterizeLayer`)

Bake a layer's full appearance ‚ÄĒ content + filters + adjustments + mask +
styles, plus composite-time text anchor / rotation transforms ‚ÄĒ into a
single canvas-resolution `PXLayer.raster`. Layer offset / opacity / blend
are preserved so the layer's relationship to the rest of the doc is
unchanged. Backed by `LayerRenderer.bakedImage(layer:canvasSize:)` which
walks the same per-layer render pipeline `composite()` uses.

Surfaces: Layer menu (‚Ćė‚áßE), LayersPanel context menu, ControlServer
`rasterizeLayer` action.

### Paint engine (`Tools/PaintEngine.swift`)

Stamp-based brush. Each pointer sample drops a soft round dab, and the
gesture handler interpolates evenly-spaced stamps along the segment from
the previous sample (10% of brush size for a smooth, continuous stroke).

Two CGContexts per stroke:

- `baseImage`: the layer's pixels at stroke start. Captured once,
  immutable for the life of the stroke.
- `strokeCtx`: scratch buffer where stamps accumulate with alpha = `flow`.

`commitToLayer()` composites `base + strokeImage` at global alpha = `opacity`,
which is what gives flow vs. opacity their distinct Photoshop meanings:

- **Flow** = per-stamp build-up rate within a stroke. Stamps overlap in the
  scratch buffer, so a low-flow stroke fades in as you drag.
- **Opacity** = the stroke's maximum alpha, applied as a single global
  multiplier at commit time. A 30% opacity stroke caps at 30% no matter
  how many stamps overlapped.

Eraser is the same accumulation model with `destinationOut` blend at commit.

Smoothing is a 1-pole low-pass filter on the input pointer:
`smoothed = lastSmoothed + (1 ‚ąí smoothing) ¬∑ (raw ‚ąí lastSmoothed)`.
`endStroke()` runs the filter against the final raw position 24√ó so the
stroke catches up to where the user actually let go.

### Brush-radius cursor

The system NSCursor scales with macOS's accessibility cursor settings and
caps around 256pt ‚ÄĒ so an NSCursor-image-based brush ring is unreliable
across the full size range (it grew at small sizes and shrank at large).
Per Apple-AppKit guidance for dynamic paint cursors, the system cursor is
just a crosshair (precise hot-spot indicator) and the *variable-size* ring
is drawn inside the SwiftUI Canvas at the live pointer location. Ring
diameter = `brush.size * docToView`, identical to the math driving the
actual stamp, so the two are guaranteed to match at any size or zoom.

Pointer tracking goes through an `NSEvent.addLocalMonitorForEvents` for
both `.mouseMoved` and `.leftMouseDragged`, so the ring keeps following
the cursor during paint strokes (where SwiftUI's hover gestures don't fire).

### Selection model

Single canonical state on `DocumentStore`:

- `selectionPath: CGPath?` ‚ÄĒ the doc-top-down outline. All selection-aware
  features (paint clipping, generative fill, etc.) read this.
- `selectionRect: CGRect?` ‚ÄĒ the path's bounding box, kept for legacy
  callers (Generative Fill, etc.) that only understand rects.
- `setSelection(rect:)` / `setSelection(path:)` / `clearSelection()` ‚ÄĒ
  always use these, never write the fields directly.

### Selection tools

| Tool             | How it builds the path                                                            |
|------------------|-----------------------------------------------------------------------------------|
| Marquee          | Drag a rect ‚Üí `CGPath(rect:)`                                                     |
| Lasso            | Drag a polyline ‚Üí close ‚Üí `CGMutablePath`                                         |
| Polygonal Lasso  | Click vertices ‚Üí double-click or click start ‚Üí close. Esc cancels in-flight.      |
| Magic Wand       | Flood-fill from the click within `magicWandTolerance` (sRGB Euclidean), Vision-extracted contour |
| Smart Select     | Vision `VNGenerateForegroundInstanceMaskRequest`, iterate instances to find which one covers the click |

All of them feed `setSelection(path:)`.

### `SelectionTools.swift`

Three building blocks:

- `floodFill(in:seed:tolerance:contiguous:)` ‚Üí grayscale CGImage. BFS from
  the seed across pixels matching seed color within tolerance. Returns a
  binary mask in canvas-resolution.
- `maskToPath(_:canvasSize:)` ‚Üí CGPath. Vision `VNDetectContoursRequest`
  produces normalized [0,1] contours; we flip Y and scale to doc top-down
  pixels.
- `refineEdge(_:radiusPx:canvasSize:)` ‚Üí CGPath. Rasterizes the input
  path, runs CIMorphology max (expand) or min (contract), re-extracts via
  `maskToPath`. Used by the Select ‚Üí Refine Edge menu.

Coordinate-systems pitfall worth flagging for future work: CGContext draw
operations use bottom-up CG coords (y=0 is the bottom), but the byte
buffer is laid out top-down in memory (offset 0 = top row). Doc-top-down
y maps directly to memory row. Earlier versions of `floodFill` confused
the two (`seedRow = h - 1 - sy`) and sampled the wrong pixel ‚ÄĒ now fixed
with a regression test (`floodFillSelectsCorrectRegion`).

### Refine Edge

`Select ‚Üí Refine Edge ‚Üí Expand/Contract 1/5/10 px` morphologically grows
or shrinks the active selection. Real feathering requires an alpha-mask
selection model; we deferred that since none of our selection-aware
features (paint clip, generative fill mask) consume soft selections yet.
A future v0.6 could add `selectionMask: CGImage?` alongside `selectionPath`
for true Photoshop-style feathered selections.

### ControlServer additions

- `rasterizeLayer` ‚ÄĒ bake the active (or `id`-targeted) layer.
- `paintStroke` ‚ÄĒ drive a paint or erase stroke headlessly. Accepts
  `points`, `color` (`#rrggbb`), `size`, `hardness`, `opacity`, `flow`,
  `smoothing`, `eraser`. Auto-creates a paint layer if the active layer
  isn't a flat raster (pencil only; eraser no-ops without a target).

These are the surface for snapshot-style integration tests and for
agent-driven QA flows.

## Tests

10 algorithmic tests in `TiramisuTests/`:

- `PaintEngineTests` (6): pencil stamp paints fg + edge transparent;
  eraser drops alpha + outside untouched; selection rect clips paint;
  free-form path clips to its actual shape (diamond polygon, not bbox);
  rasterize preserves composite output ‚Č§ 2% mean abs delta; rasterize
  clears bake-able state.
- `SelectionToolsTests` (4): floodFill seeds and selects the correct
  region (regression for the orientation bug); non-contiguous floodFill
  picks all matches; maskToPath round-trips a rect within ~5% area;
  refineEdge expand‚Üícontract returns near the original bbox.

Vision-based features (Smart Select) deliberately don't have unit tests
‚ÄĒ the model isn't deterministic enough to make them stable. Manual smoke
test on a multi-subject photo serves as acceptance.

## Deferred to v0.6+

- **Magnetic Lasso** (live-wire / Mortensen & Barrett 1995) ‚ÄĒ interesting
  but multi-day. Mostly obsoleted by Smart Select for typical workflows.
- **SAM2 via Core ML** for sub-instance / sub-part selection. The natural
  next AI upgrade; click-anywhere-on-X for any X, including backgrounds
  and surfaces Vision instance segmentation can't pick out.
- **Soft / feathered selections** via `selectionMask: CGImage?`.
- **Brush dynamics from pressure / tilt** (when tablet input lands).
- **Brush presets**.
